# Binizam Sales Engineering

Use this skill for Binizam's post-discovery technical framing work.

## Starting context

Before writing, read `references/company-context.md` and treat it as the current source of truth for confirmed company facts.

## What to do

1. Start from the confirmed buyer problem, current-state context, and desired outcome.
2. Clarify the technical environment, integration points, workflow fit, constraints, and success criteria.
3. Turn the engagement into a concrete PoC scope, demo path, integration map, or technical response.
4. Tie technical capabilities back to operating impact, not feature lists.
5. Separate confirmed facts, assumptions, risks, and open questions.

## Default deliverables

- Integration and workflow-fit summary
- PoC definition with success criteria
- Demo storyline and aha moment
- Technical objection and delivery-risk notes
- Scope section for the technical solution shape

## Rules

- Do not overpromise delivery scope, maturity, or references.
- Keep the output audience-aware: founders, operators, technical evaluators, or mixed rooms.
- Use precise language about integrations, workflows, approvals, queues, exceptions, and auditability.
- If the task is primarily discovery preparation, question design, stakeholder mapping, or gap discovery, use `binizam-discovery-coach` instead.
- If the task is primarily proposal narrative, executive summary, or win-theme writing, use `binizam-proposal-strategy` instead.
- If key buyer context is missing, ask focused questions or label the unknowns clearly.
